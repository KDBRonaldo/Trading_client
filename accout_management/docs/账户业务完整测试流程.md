# 账户业务完整测试流程

本文档是当前可交付版本的统一测试入口，覆盖功能、校验、联调、并发、安全与页面回归。执行时以本文为准，之前分散的测试方案、数据集、执行流程、问题整理文档已合并。

## 1. 测试目标

- 验证工作人员端、投资者端、后端接口、数据库落库的一致性
- 验证本轮新增与修复功能已经形成闭环
- 验证外部资金/持仓回调、管理员冻结接口、年度结息接口可联调
- 验证并发、越权、令牌失效、SQL 注入、XSS 等风险点

## 4. 执行顺序

1. 启动环境与健康检查
2. 登录与鉴权回归
3. 首次登录证书认证
4. 证券账户主链路

## 5. 执行前检查

- 打开 `/login`，确认账号和密码输入框为空
- 确认页面不再显示测试账号提示文案
- 直接访问 `/`、`/funds`、`/securities`、`/user`，应自动跳回 `/login`
- MySQL 登录后执行：

```sql
USE account_db;
SELECT COUNT(*) FROM staff;
SELECT * FROM login_certificate_state;
```

预期：

- `staff` 至少有基础工作人员数据
- 初始时 `login_certificate_state` 为空或只有已测账号记录

## 6. 功能测试链路

### 6.1 登录与鉴权回归

步骤：

1. 打开 `http://localhost:5173/login`
2. 点击“工作人员”登录入口
3. 输入 `staff01 / 123456`
4. 点击“登录”

预期：

- 首次不会直接进入主页，而是跳转 `/certificate`
- 右上角不会预先显示 `Admin`

数据库核对：

```sql
SELECT * FROM login_certificate_state WHERE subject_type='STAFF' AND subject_key='staff01';
```

### 6.2 首次登录证书认证

步骤：

1. 在证书认证页输入 `CERT-123456`
2. 点击“确认认证”
3. 进入管理端后刷新页面
4. 再次访问管理端业务页

预期：

- 认证后成功进入管理端
- 刷新页面后仍保留登录态
- 再次登录同一账号时不再强制要求证书认证

数据库核对：

```sql
SELECT subject_type, subject_key, certificate_verified, verified_at
FROM login_certificate_state
WHERE subject_type='STAFF' AND subject_key='staff01';
```

### 6.3 个人证券账户开户

页面与按钮：

- 管理端左侧点击“证券账户业务”
- 点击“开设证券账户”
- 填写表单后点击“确认开户”

预期：

- 开户成功
- 列表中出现新证券账户
- 证件号列表页应做脱敏显示
- 操作日志新增“证券开户”

数据库核对：

```sql
SELECT * FROM investor ORDER BY investor_id DESC LIMIT 1;
SELECT * FROM security_account ORDER BY open_date DESC;
SELECT * FROM operation_log ORDER BY log_id DESC LIMIT 5;
```

### 6.4 个人代办开户

用例：

- 代办人信息完整，开户成功
- 缺少代办人证件号，开户失败
- 选择“本人办理”时，不强制填写代办人信息

预期：

- 代办成功场景下，`investor.agent_name` 与 `agent_id_number` 落库
- 缺证场景返回参数或业务错误，不写脏数据

### 6.5 法人证券账户开户

页面与按钮：

- “证券账户业务” -> “开设证券账户”
- 投资者类型选择“法人”
- 填写法人完整资料 -> “确认开户”

预期：

- “法人姓名”字段命名正确
- 法人开户完整跑通
- 后续可继续执行法人挂失、补办、资料修改、销户

数据库核对：

```sql
SELECT type, name, legal_number, business_license, authorize_name, executor_name
FROM investor
WHERE type='法人'
ORDER BY investor_id DESC;
```

### 6.6 资金账户开户与绑定

页面与按钮：

- 管理端左侧点击“资金账户业务”
- 点击“开设资金账户”
- 手动输入证券账户号、证件号、交易密码、取款密码
- 点击“确认开户”

预期：

- 必须先有证券账户才能开户
- 资金账户创建后自动绑定证券账户
- 不再出现“扫描或输入证券账户卡号”这类误导性描述

数据库核对：

```sql
SELECT fund_acc_no, sec_acc_no, available_balance, frozen_balance, status
FROM fund_account
ORDER BY open_date DESC;

SELECT sec_acc_no, linked_fund_acc, status
FROM security_account
ORDER BY open_date DESC;
```

### 6.7 黑名单校验

场景：

- 黑名单命中：禁止开户
- 黑名单不可达：warning 后默认放行

验证方式：

- 使用命中数据开户
- 暂停或屏蔽黑名单服务后再次开户

预期：

- 命中时直接失败
- 不可达时后端日志有 warning，但开户流程继续

### 6.8 工作人员资金操作

页面与按钮：

- “资金账户业务”列表
- 使用“存款”“取款”“改密”“流水”“挂失”“补办”“销户”等按钮

重点检查：

- 取款不再有无意义的 `+1/-1` 按钮
- 改密必须校验旧密码
- 资金流水弹窗可展示最近 50 条记录
- 流水中如关联股票，应显示股票名称、代码、股数变化量

数据库核对：

```sql
SELECT * FROM fund_transaction_log ORDER BY log_id DESC LIMIT 20;
```

### 6.9 投资者端登录、查询、改密

步骤：

1. 在 `/login` 切换到投资者登录
2. 输入资金账户号和交易密码
3. 首次同样完成证书认证
4. 进入“我的账户”
5. 查看资金快照、证券快照
6. 进入“修改密码”
7. 输入旧密码、新密码、确认新密码后提交

预期：

- 登录成功后不会再报 `authToken: 不能为空`
- 页面刷新后仍保留有效登录态
- 余额展示和最大可转出金额同步
- 密码显示小眼睛状态应稳定可控，不应一闪即失

### 6.10 证券账户挂失、补办、销户

挂失步骤：

1. 进入“证券账户业务”
2. 点击目标账户“挂失”
3. 工作人员手动输入证件号
4. 输入原因后提交

预期：

- 挂失时必须校验证件号
- 账户状态变为冻结
- 持仓全部转入冻结持仓

补办步骤：

1. 在冻结账户栏点击“补办”
2. 按开户级别重新填写完整资料
3. 提交

预期：

- 旧证券账户转已销户
- 新证券账户生成
- 原持仓迁移到新证券账户
- 如是因资金账户挂失导致的冻结，则需待资金补办后才可恢复

销户前置条件：

- 有持仓不能销户
- 有冻结持仓不能销户
- 清仓后才允许销户

### 6.11 资金账户挂失、补办、销户

挂失步骤：

1. 在“资金账户业务”点击“挂失”
2. 手动输入身份证号
3. 提交

预期：

- 可用余额全部转入冻结余额
- 证券账户同步冻结
- 持仓全部同步冻结
- 账户管理区域只保留“补办”，不应保留“修改密码”钥匙图标

补办步骤：

1. 点击“补办”
2. 像开户一样输入完整个人信息
3. 输入新交易密码、新取款密码
4. 提交

预期：

- 新密码输入框为掩码显示
- 老资金账户转已销户，余额清零
- 新资金账户继承原总余额并解冻
- 老证券账户若需换号则自动重绑定新账户
- 持仓从冻结恢复为可交易持仓

销户前置条件：

- 有可用余额不能销户
- 有冻结余额不能销户
- 提现清零后才允许销户
- 销户后账户不应再出现在正常/冻结列表中

### 6.12 无资金账户冻结与重新绑定恢复

步骤：

1. 关闭资金账户
2. 查看证券账户状态
3. 重新开立或绑定新的资金账户
4. 再查看证券账户状态

预期：

- 无资金账户时证券账户进入 `NO_FUND_FROZEN`
- 重绑成功后自动恢复为正常

### 6.13 账户展示与 Dashboard 回归

检查项：

- Dashboard 总数不包含已销户账户
- 正常账户与冻结账户分栏展示
- 已销户账户不再出现在账户业务列表
- 操作记录列表与数据库 `operation_log` 对齐

数据库核对：

```sql
SELECT status, COUNT(*) FROM security_account GROUP BY status;
SELECT status, COUNT(*) FROM fund_account GROUP BY status;
SELECT * FROM operation_log ORDER BY log_id DESC LIMIT 20;
```

## 7. 外部联调测试

### 7.1 投资者资金快照

- 接口：`GET /api/external/fund/snapshot`
- 预期：资金账号、可用余额、冻结余额与数据库一致

### 7.2 投资者证券快照

- 接口：`GET /api/external/security/snapshot`
- 预期：持仓数量、冻结数量、可用数量与 `holding` 表一致

### 7.3 外部资金回调

- 接口：`POST /api/external/trade/fund-balance`
- 场景：
  - 买入冻结
  - 买入扣款
  - 卖出回款
  - 撤单解冻
- 预期：余额变化正确，生成资金流水

### 7.4 外部持仓回调

- 接口：`POST /api/external/trade/security-holding`
- 场景：
  - 买入增加
  - 卖出冻结
  - 卖出扣减
  - 撤单释放
- 预期：持仓数量与冻结数量变化正确，生成持仓变动日志

### 7.5 幂等重放

同一 `ref_order_id` 重放多次：

- 资金回调只入账一次
- 持仓回调只生效一次
- 接口返回 `duplicate=true`

## 8. 年度结息测试

接口：

- `POST /api/admin/fund/settle-annual-interest`

检查项：

- 符合条件账户才参与结息
- 结息后 `available_balance` 增加
- `last_interest_date` 更新
- 生成 `txn_type=结息` 的资金流水

数据库核对：

```sql
SELECT fund_acc_no, available_balance, last_interest_date, annual_interest_rate
FROM fund_account
ORDER BY fund_acc_no;

SELECT * FROM fund_transaction_log
WHERE txn_type='结息'
ORDER BY log_id DESC;
```

## 9. 页面回归测试

- 登录页默认值为空
- 登录页不显示测试提示小字
- 未登录直达业务页会被拦截
- 管理端右上角显示真实用户名，不显示固定 `Admin`
- 工作人员端与投资者端“切换”按钮位置统一
- 切换后必须清空当前登录态并回到登录页
- 冻结账户、正常账户分栏显示
- 补办/挂失表单完整性符合业务要求
- 修改密码必须验证老密码
- 余额卡片、流水页、Dashboard 统计值相互一致

## 10. 安全测试

- 错误 token 访问受保护接口，应返回鉴权失败
- 被停用工作人员的旧 token 不可继续使用
- 投资者 token 不可访问工作人员接口
- SQL 注入字符串不能绕过登录、身份校验、绑定校验
- XSS 输入应被当作普通数据返回 JSON，不应服务端执行
- 页面刷新保留会话，但退出登录后旧 token 不应再被复用

## 11. 并发与稳定性测试

并发场景：

- 多工作人员同时开户
- 同一资金账户多次重复资金回调
- 同一证券账户多次重复持仓回调
- 不同工作人员同时查询操作日志、资金流水

预期：

- 无死锁、无脏写
- 幂等回调最终只落一笔有效记录
- 列表页与数据库最终一致

## 12. 证据留存要求

每条关键链路至少保留：

- 页面截图
- 请求参数
- 接口返回
- 数据库 SQL 与查询结果
- 如有异常，附后端日志关键行

## 13. 通过标准

一条用例只有同时满足以下条件才算通过：

- 页面行为正确
- 接口返回正确
- 数据库状态正确
- 没有产生重复数据、脏数据或越权数据
